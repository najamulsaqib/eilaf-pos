# eilaf-pos Project Standards

This project is an Electron desktop POS application with React frontend, local SQLite storage for business data, and Supabase for authentication only.

## Architecture Rules

### 1. Component Organization

**ALWAYS use existing components from the components folder. Never create new components in pages.**

Available component categories:

- `components/ui/` - Button, TextField, SelectField, CheckboxField, Card, Chip, IconButton, DropdownMenu, DropZone, Modal, ConfirmDialog
- `components/common/` - EmptyState, ServiceCard, LoadingSpinner, StatCard, FloatingActionBar
- `components/layout/` - Sidebar, Header, AppLayout, TabBar
- `components/table/` - DataTable, Pagination

**Rules:**

- ALWAYS check existing components before creating new ones
- New components MUST be created in the appropriate subfolder under `src/renderer/components/`
- Components should be reusable and generic
- Use PascalCase for component file names (e.g., `SalesCard.tsx`)

### 2. Data Access Boundaries (Critical)

**NO direct database calls in pages or components. NEVER import `better-sqlite3` in renderer code.**

Required boundary:

1. Renderer UI (`pages`, `components`) uses hooks/services only
2. Hooks/services call preload API (`window.electron.*`) only
3. Main process IPC handlers call repositories
4. Repositories in `src/main/db/` access SQLite (`better-sqlite3`)

**Rules:**

- All POS business data (products, invoices, customers, inventory, payments, reports, settings) must be persisted in local SQLite
- All SQLite access belongs to `src/main/db/` and must be exposed through typed IPC contracts
- Keep SQL and schema concerns in repository files, not in IPC handlers
- Use transactions for multi-step writes that must be atomic

### 3. Supabase Usage (Auth Only)

**Supabase is restricted to authentication workflows only.**

Allowed:

- Sign in
- Sign out
- Session restore/refresh
- Forgot password and password reset

Not allowed:

- Business table CRUD through Supabase
- POS transactional data in Supabase
- Supabase realtime for business records

### 4. Hooks Pattern

**ALL renderer data fetching/mutations MUST go through hooks in `src/renderer/hooks/`.**

**Rules:**

- Hooks use React Query (@tanstack/react-query) for cache and async state
- Business-data hooks call renderer services that use IPC
- Auth hooks/context may use Supabase auth client APIs
- Use camelCase for hook file names (e.g., `useSales.ts`, `useInventory.ts`)
- Hooks should encapsulate loading, error handling, and mapping to UI-friendly models

### 5. Pages Structure

**MUST use folder-based routing, NOT file-based routing.**

Correct structure:

```text
pages/
  dashboard/
    index.tsx
    components/
      SalesStats.tsx
  sales/
    index.tsx
    new/
      index.tsx
    [id]/
      index.tsx
      edit/
        index.tsx
    components/
      SalesTable.tsx
```

**Rules:**

- Each page MUST have its own folder with `index.tsx`
- Sub-pages go in nested folders with their own `index.tsx`
- Use kebab-case for folder names (e.g., `sales-history/`, `inventory-adjustments/`)
- Page-specific components go in a `components/` subfolder within the page folder

### 6. Naming Conventions

- **Folders:** kebab-case (`inventory-management`, `sales-history`)
- **Hooks:** camelCase (`useSales.ts`, `useInventory.ts`)
- **Variables:** camelCase (`saleTotal`, `selectedCustomer`, `isLoading`)
- **Components:** PascalCase (`SalesCard.tsx`, `InventoryTable.tsx`)
- **Constants:** UPPER_SNAKE_CASE (`MAX_RECEIPT_ITEMS`, `DEFAULT_PAGE_SIZE`)
- **Types/Interfaces:** PascalCase (`SaleRecord`, `InventoryItem`, `CustomerProfile`)

### 7. Internationalization (i18n) Standards

**ALL visible UI strings must be translated. Never hardcode text in JSX.**

```tsx
// WRONG
<h1>Dashboard</h1>

// CORRECT
const { t } = useTranslation();
<h1>{t('dashboard.title')}</h1>
```

**Rules:**

- Use `useTranslation()` from `react-i18next` in every component that renders text
- Add every new string to **both** `src/renderer/i18n/locales/en.json` and `src/renderer/i18n/locales/ur.json`
- Keys are namespaced by feature: `nav.*`, `dashboard.*`, `settings.*`, `common.*`
- Use `common.*` keys for shared labels (`common.save`, `common.cancel`, `common.delete`, etc.)
- Keep keys in camelCase and nest by feature, not by component

**RTL layout — use Tailwind logical properties:**

| Avoid (physical) | Use (logical)   |
| ---------------- | --------------- |
| `ml-*` / `mr-*`  | `ms-*` / `me-*` |
| `pl-*` / `pr-*`  | `ps-*` / `pe-*` |
| `border-l` / `border-r` | `border-s` / `border-e` |
| `left-*` / `right-*` | `start-*` / `end-*` |
| `text-left`      | `text-start`    |

Physical directional classes (`ml-*`, `mr-*`, `text-left`, etc.) are **banned** — they break RTL layout.

For icons that must flip in RTL, use the `rtl:` variant:

```tsx
<ChevronRightIcon className="rtl:rotate-180" />
```

**Locale access:**

```tsx
import { useLocale } from '@contexts/LocaleContext';
const { locale, setLocale, isRTL } = useLocale();
```

**Font:** Urdu locale automatically uses Jameel Noori Nastaleeq Kasheeda via `:root:lang(ur)` in `styles.css`. No per-component font changes needed.

**Full documentation:** [`.github/copilot/skills/i18n.md`](.github/copilot/skills/i18n.md)

### 8. UI Library Standards

**ONLY use approved UI libraries:**

- **Icons:** ONLY use `@heroicons/react`
- **UI Primitives:** Use app UI components from `components/ui/`
- **Styling:** TailwindCSS only (no CSS modules, no styled-components)
- **Notifications:** Use `sonner` for toasts

**Rules:**

- **Never use a native `<select>` element.** Always use `SelectField`
- **Never use a raw `<button>` for actions.** Use `Button`, `IconButton`, or `DropdownMenu`
- All components are default exports
- Use existing UI components before creating new ones
- All styling must use Tailwind utility classes
- Responsive design must be mobile-first

### 9. Theming Standards

eilaf-pos uses **CSS custom property–based semantic tokens**. All theme logic lives in `src/renderer/styles.css`.

**Rules:**

- **Never write `dark:` variants in component or page files.** Define dark (or any theme) overrides in `styles.css` under the appropriate theme class block (`.dark { … }`).
- **Never use hardcoded Tailwind color classes** (`bg-white`, `text-slate-900`, `border-gray-200`, `blue-*`) in components. Use semantic tokens instead.
- Use `bg-surface` / `bg-surface-raised` / `bg-surface-muted` for all backgrounds.
- Use `text-ink` / `text-ink-dim` / `text-ink-faint` / `text-ink-ghost` for all text.
- Use `border-edge` / `border-edge-strong` / `border-edge-muted` for all borders.
- Use `ring-focus-ring` / `border-focus-ring` for keyboard focus rings — never `ring-primary-500` directly.
- Use `primary-*` (brand palette) for CTAs and accents — it is a fixed palette, not overridden per theme.
- For new accent colors needed in components, add `--color-*` variables to `@theme` in `styles.css` and their dark override to `.dark { }`.

**Full token reference:** [`.github/copilot/skills/tailwindcss.md`](.github/copilot/skills/tailwindcss.md)

### 9. Code Quality Standards

- Write clean, readable, self-documenting TypeScript
- Avoid `any` unless there is a strong justification
- Keep functions small and focused
- Extract reusable logic into `src/renderer/lib/` or `src/main/db/` utilities
- Add comments only for non-obvious business logic
- Handle errors with user-friendly messaging
- Validate user input before mutation
- Implement loading, empty, and error states

### 10. SQLite Standards

- Keep schema management in `src/main/db/` (migrations/init)
- Use prepared statements; avoid string-concatenated SQL
- Use transactions for write batches and cross-table updates
- Return normalized row models from repositories
- Convert SQLite errors to friendly domain errors in repositories/services
- Keep repository methods focused (`create`, `list`, `getById`, `update`, `delete`)

### 11. IPC Standards

- IPC channel names must be explicit and namespaced (e.g., `sales:list`, `inventory:updateStock`)
- Validate payloads before DB calls
- Do not expose generic raw SQL IPC channels
- Keep preload API typed in `src/renderer/preload.d.ts` and aligned with handlers
- Never leak Node internals into renderer

## File Structure Reference

```text
src/
  main/
    db/                 # SQLite client, migrations, repositories
    ipc/                # IPC handlers wrapping repository methods
    preload.ts          # Context bridge API

  renderer/
    components/         # Reusable components only
      ui/
      common/
      layout/
      table/
    hooks/              # Custom hooks (UI data boundary)
    services/           # Renderer-side IPC API wrappers
    pages/              # Folder-based pages
    lib/                # Renderer utilities
    contexts/           # Auth/session contexts (if used)

  shared/               # Shared contracts/types between main and renderer
```

## Migration Priorities

1. Move business CRUD from Supabase calls to SQLite repositories + IPC
2. Keep Supabase usage only inside auth flows
3. Ensure renderer pages/components only use hooks/services
4. Standardize page routing to folder-based `index.tsx` files
5. Enforce approved UI primitives and naming conventions
